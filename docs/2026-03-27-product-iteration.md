# Watcher 产品迭代文档

- 日期：2026-03-27
- 迭代主题：历史数据管理细化、视频分析任务泛化与流式链路完善、模型输出容错增强、底部共享导航交互修复
- 适用范围：截至 2026-03-27 当日已落地代码

## 1. 本次迭代目标

本轮迭代不再只做界面层微调，而是同时推进以下四条主线：

1. 将第四页“历史数据管理”从仅能浏览记录，升级为可直接回看任务媒资、查看分段、查看监控证据的统一工作台。
2. 将视频分析任务统一为“自然语言需求 + 模板任务”共用的一套泛化流程，避免两套逻辑并行演化。
3. 为视频分析补齐流式输出与分片处理链路，使单段分析、阶段反馈与最终汇总处于同一状态模型中。
4. 提升模型输出的鲁棒性，尤其修复 `confidence` 返回中文等级时导致的视频分段分析失败问题。

## 2. 历史数据管理升级

### 2.1 第四页形成统一历史工作台

当前第四页已形成统一的历史管理结构，包含三部分：

- 历史存储概览：展示记录数、媒体数、占用体积。
- 统一时间流：将“视频分析”和“实时监控”按时间顺序混排展示。
- 记录详情：根据选中记录展示不同类型任务的完整结果与关联媒体。

配套的数据结构已统一抽象为：

- `HistoryRecordItem`
- `HistoryRecordSelection`
- `HistoryRecordDetail`
- `VideoHistoryDetail`
- `MonitorHistoryDetail`

这使历史页不再是单一功能页，而是一个可持续扩展的归档工作台。

### 2.2 视频分析历史详情可直接按片段回看

视频分析记录详情已补齐以下能力：

- 展示任务摘要、最终结论、时间线事件。
- 展示每个视频分段的本地视频路径与阶段摘要。
- 点击某一段时，只在该分段卡片下方展开对应播放器，而不是固定在顶部统一播放。
- 切换到其他页面时自动关闭播放器，减少资源占用。

在后续细化中，又进一步修复了两个交互问题：

- 去掉分段卡片展开时大面积灰色点击波纹，避免覆盖卡片与视频区域。
- 将“展开/收起”的点击区限制在分段信息区，播放器区域保留给暂停、快进、快退等原生控制交互，不再因为再次点击而误收起。

### 2.3 实时监控历史记录补齐完整证据链

实时监控历史详情当前已经支持展示：

- 完整会话视频
- 基准图片
- 关键事件截图
- 附加快照
- 运行统计

这使实时监控记录不再只是文字日志，而是具备可回放、可复核、可追溯的证据链结构。

## 3. 视频分析流程统一

### 3.1 模板任务与自然语言任务统一为一套流程

本轮对视频分析的关键重构，是将任务入口统一为同一个草稿/执行模型，而不是将模板任务与自然语言任务分开实现。

当前统一思路为：

- 模型解析自然语言需求，输出任务类别、执行参数、片段分析提示词、全局汇总提示词。
- 模板任务本质上也是一组预设草稿，最终仍进入同一条确认与执行链路。
- 用户仍可在确认执行参数阶段手动调整 FPS、片段时长、采样间隔、总时长等参数。

这样做的收益：

- 模板与自由输入不再形成两套状态分叉。
- 后续只需持续强化“任务规划”这一层即可。
- UI、Repository、持久化模型都能围绕同一套字段演进。

### 3.2 视频分析统一为“分片 + 汇总”范式

当前视频分析已明确收敛为统一的工作范式：

- 按配置周期录制片段
- 单段录完后立即上传/预处理/分析
- 分段结果持续回流到运行状态
- 全部片段结束后再执行全局汇总

该范式可覆盖：

- 短时密集观察
- 长时稀疏记录
- 第三视角记录类模板任务
- 用户自然语言描述的动态采样需求

对应地，系统中已明确区分两类提示词：

- `segmentAnalysisPrompt`：指导单个片段如何分析
- `finalSummaryPrompt`：指导如何整合全部片段形成最终结论

二者都已要求使用可供用户直接编辑的简体中文。

### 3.3 任务规划输出进一步结构化

任务规划模型当前除了基础执行参数外，还会输出：

- `templateId`
- `templateLabel`
- `taskCategory`
- `segmentAnalysisPrompt`
- `finalSummaryPrompt`
- `autoStartStreamingOutput`

这为后续基于“时长 + 语义”的意图解析继续增强提供了稳定接口。

## 4. 流式输出与状态回流

### 4.1 视频分析支持可选流式输出

当前视频分析已具备流式输出开关，并接入统一的运行状态：

- 运行状态中可携带 `streamingEnabled`
- 运行状态中可携带 `streamingBuffer`
- UI 已能在视频分析页面显示流式返回内容

这使模型在分段分析或汇总阶段的文本输出，不再只能在任务结束后一次性呈现。

### 4.2 流式链路已与分段执行打通

当前分段分析与最终汇总均已支持：

- 非流式调用
- 流式调用
- 将流式回调中的累积文本实时写回状态

对应的统一逻辑已经沉淀在视频分析仓储层中，而不是散落在界面层。

### 4.3 现阶段的边界

虽然已有流式文本返回，但当前模型侧仍是“片段录制结束后再分析对应片段”，并不等同于“录制过程中持续理解尚未完成的视频文件”。  
因此，本轮已落地的是“分段流式反馈”，不是“同一段视频边录边分析”的真正在线理解。

## 5. 模型输出容错与解析增强

### 5.1 修复 `confidence` 中文等级导致的分段失败

此前在流式视频分析中，模型若返回：

- `"confidence": "高"`

会在解析阶段触发数值转换异常，进而报出：

- `failed to analyze the video segment`

当前已通过“提示词约束 + 解析容错”双重方案修复：

- 数值 `0~1` 仍为首选格式
- 允许 `"高" / "中" / "低"`
- 允许英文等级
- 允许百分数字符串
- 非法值时不再中断整段结果，而是保留事件并将 `confidence` 置空

### 5.2 解析器升级为宽容读取

`ModelOutputParser` 已增强：

- `confidence` 字段从固定 `Float?` 扩展为更宽容的原始值读取
- 统一通过容错解析函数完成归一化
- 视频分析与实时监控两条解析路径共用同类处理策略

这次改动的意义是：不再把模型输出轻微偏差直接升级为任务失败。

### 5.3 提示词同步约束输出形态

以下几类提示词已同步更新约束：

- 视频片段分析提示词
- 视频全局汇总提示词
- 视频任务规划提示词
- 模板任务默认提示词
- 实时监控 JSON 输出提示词

新增要求包括：

- 输出字段值优先使用简体中文
- `timelineEvents` 项必须包含 `timestampSeconds`、`title`、`detail`、`confidence`
- `confidence` 优先返回 0 到 1 的数字，无法量化时允许返回“高”“中”“低”

## 6. 底部共享导航继续修复

### 6.1 偶发“灰色选中块卡住”问题定位

在底部透明悬浮导航栏中，偶发会出现灰色选中块停留在错误页面的问题。  
当前已定位到高概率原因：

- 页面主体动效使用的是连续的 `pagerPosition`
- 导航选中块此前单独使用 `currentPage + currentPageOffsetFraction`
- 点击切页时的早退判断只看 `currentPage`

当切页动画被打断、连续点击或页面尚未完全落稳时，这几套状态会短暂失配，导致选中块偶发卡住。

### 6.2 已落地修复

当前已完成以下收口：

- 共享底部导航的选中块直接绑定 `pagerPosition`
- 不再由导航组件自行重算独立位置
- 点击切页的早退判断改为基于“当前位置与目标位置距离”，而不是只看 `currentPage`

这使导航栏的选中块与整体页面切换动画使用同一条连续状态源。

## 9. 后续建议

若继续沿当前方向优化，建议优先关注：

1. 将“录制中实时反馈”从“分段完成后流式返回”进一步推进到更细粒度的切片调度体验。
2. 继续强化自然语言意图解析中“时长 + 语义”到采样策略的映射质量。
3. 在历史详情页中为视频片段增加更明确的展开/收起视觉控件，进一步降低误触成本。
4. 对底部导航切页增加串行化或节流保护，彻底消除高频连续点击带来的状态竞争。
